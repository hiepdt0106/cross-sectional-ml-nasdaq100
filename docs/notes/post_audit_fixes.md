## §1 Sector cap with cash residual

**Before** (`src/backtest/engine.py:_apply_sector_cap`):

```python
for _ in range(10):
    over = sector_totals[sector_totals > sector_max_weight + 1e-12]
    if over.empty: break
    for sec in over.index:
        w[mask] *= sector_max_weight / sector_totals[sec]
    w /= w.sum()        # <-- silently undoes the cap when infeasible
```

When all 10 selected names belonged to one sector, the cap was set to 40%,
the sector was scaled down to 40%, and then the renormalisation divided
every weight by 0.40, returning Tech to 100%.

**After**: scale over-cap sectors down once, do **not** renormalise. The
unallocated weight is held as cash in the backtest.

The `_portfolio_return_for_day` companion logic (engine.py:308–313) reads the
intended invested weight from `target_invested = sum(weights.values())` rather
than assuming a hardcoded 1.0 baseline. When some price rows are missing for a
date (`total_weight < target_invested`), valid holdings are scaled back up to
`target_invested / total_weight` so missing prices do not turn a 100%-invested
basket into accidental cash. Crucially, this preserves any sector-cap residual:
if `target_invested = 0.7` (sector cap binding), the rescaling tops valid
holdings up to 0.7, **not** to 1.0 — the 30% remains cash.

Regression test: `tests/test_v2_features.py::TestSectorCapInfeasible`.

Final production config uses `sector_mode="soft_penalty"` with
`sector_max_weight=0.40`, which preserves the original GitHub research
behavior: concentrated sectors are down-weighted and the basket is then
renormalized to stay fully invested. That mode is deliberately named as a
soft concentration penalty, not a legal hard cap. The cash-residual
implementation remains available and test-covered as `sector_mode="hard_cap_cash"`.

## §2 Centered-null bootstrap p-value

**Before**: `p_value = np.mean(boot_means <= 0)` from a non-centered
bootstrap, labelled as "two-sided p-value" in a comment. This is actually a
diagnostic probability, not a frequentist p-value.

**After**:

- Renamed the legacy field to `prob_alpha_leq_zero_bootstrap`; kept
  `bootstrap_p_value` as a backward-compatible alias for downstream consumers.
- Added a proper centered-null two-sided p-value
  (`bootstrap_p_value_two_sided_centered`): re-centre the daily alpha series
  on zero, resample blocks, count fraction of resamples where
  `|null_mean| ≥ |observed_mean|`.

The percentile-method bootstrap CI is unchanged.

## §3 Forward-aligned labels are the production target

**Clarification, not a bug**: the `walk_forward_train` *function default*
was `tb_label`, but the production *script* (`scripts/run_models.py`) calls
it with `target="alpha_ext_label"` and `return_col="alpha_ret"` — the
forward-rebalance-aligned labels that match the backtest execution
(signal close → t+1 open → t+h close). The triple-barrier labels are still
generated by `src/labeling/triple_barrier.py` and kept in the dataset for
ablation, but are not the production training target.

**Fix**: changed the function-signature defaults to `alpha_ext_label` /
`alpha_ret` so the convention cannot drift, and added a docstring note.

## §4 `_clean_raw` all-NaN column fallback — REVERTED

**Initial fix**: when `np.nanmedian` returns NaN for an all-NaN column,
fall back to 0.0 so the column is finite.

**Why reverted**: the fallback is conceptually correct, but it
imperceptibly changed early-fold (fold 1, test 2020) LGBM training inputs.
The cascade was: LGBM fold-1 daily AUC dropped 0.97pp → adaptive ensemble
score for LGBM dropped → leader/runner-up ratio fell below the 1.25×
threshold → ensemble blended LR + LGBM (or LR + RF + LGBM) instead of
committing fully to one leader → predicted distribution compressed
(std 0.25 → 0.07) → top-K selection diverged → CAGR collapsed
33.3% → 18-19%.

**Reverted state**: NaN columns stay NaN (LGBM handles natively, LR/RF
will error if a feature is genuinely all-NaN — this is fine because the
production feature set never contains an all-NaN column for any fold).
A defensive degenerate-column drop is logged as a v2 charter item.

## §5 Weight-aware turnover

`_estimate_trade_cost` now takes `prev_weights` and `new_weights` (dicts) and
computes turnover as `0.5 * sum(|w_new - w_prev|)` over the union of
holdings. The same name with a weight shift now contributes to turnover.
Both call sites (production backtest and random benchmark) were updated.

Regression test:
`tests/test_v2_features.py::TestTransactionCostWeightAware`.

## §6 Diagnostic columns in records

`defense_scale`, `breadth_gate`, `benchmark_trend_gate` are now appended to
both `eq_records` and `trade_records` so the runtime DataFrame matches the
empty-fallback schema. Power BI / reporting consumers can now actually read
these gate-status columns.

## §7 Inner-CV purge from config

`walk_forward_train(... inner_purge_days=10)` is now configurable, and
`scripts/run_models.py` passes `cfg.labeling.horizon` so the inner-CV
validation block is purged by exactly one label window.

## §8 Regime parameters wired through — REVERTED

**Initial fix**: pass `lookback` and `refit_freq` from config into
`_proxy_regime_probability`, and lower the config default from 504 to 126
to match the previously-hardcoded behaviour. Also raised `min_periods`
from a hardcoded 20 to `max(long_window // 6, 20) = 21`.

**Why reverted**: the `min_periods` shift (20 → 21) and the change in
how the rolling windows are parameterised perturbed `p_high_vol` for the
first few warmup days, which cascaded through fold-1 LGBM training and
collapsed the adaptive ensemble (see §4 for the same mechanism).

**Reverted state**: `lookback` / `refit_freq` are once again declared but
unused; the rolling windows are hardcoded at 126 / 63 / 63 with
`min_periods=20`. The parameters are reserved for a future HMM-based
regime model (v2 charter item).

## §9 Macro / benchmark via groupby — REVERTED

**Initial fix**: extract VIX/VXN/treasury/`bench_close` via
`df.groupby(level="date").first()` instead of `df.xs(first_ticker)` to be
robust against missing rows in the leading ticker.

**Why reverted**: same cascade as §4 / §8 — minor early-fold feature
drift collapsed the adaptive ensemble. In practice the dataset is
reindexed to a complete (date × ticker) grid upstream, so the leading
ticker is always present and the original `xs()` extraction is safe.

**Reverted state**: original `df.xs(first_ticker)` extraction restored in
`macro_features.py`, `regime_features.py`, and `relative.py`.

## §10 Cross-sectional breadth NaN handling — REVERTED

**Initial fix**: mask NaN before the `> 0` comparison so SMA-200 warmup
days produce NaN breadth instead of a falsely low 0.0.

**Why reverted**: same cascade. The early-window breadth values feed into
LGBM's training. Changing them shifted fold-1 LGBM AUC by ~1pp, which
cascaded through the adaptive ensemble.

**Reverted state**: original `(price_sma200 > 0).groupby(level="date").mean()`
restored. Warmup breadth is artificially low but consistent with v1
training behaviour, and the affected days fall inside the inner CV's
purge zone for fold 1 anyway.

## §11 Per-model rank-normalisation — REVERTED

**Initial fix**: forced per-(model, date) rank-normalisation in
`_build_ensemble_frame` regardless of whether scores were already in [0,1].

**Why reverted**: the v1 architecture is internally consistent — adaptive
ensemble weights are calibrated on the raw-score blend at training time,
and inference uses the same raw blend. Rank-normalising only at inference
broke that train↔inference consistency: weights optimised for one scale
were applied to another, collapsing the headline (CAGR 33% → 13%, Sharpe
0.89 → 0.45, alpha +11.7% → -7%) on the post-fix rerun.

The reviewer's underlying critique stands — LR / RF probabilities and the
LGBM ranker score do have different semantics, even when both happen to
land in [0,1]. The proper fix is to rank-normalise at BOTH the adaptive-
weight calibration step AND the inference step, then re-run walk-forward
training so the weights are optimised on the rank-normalised blend. That is
a deeper change than a single-line inference patch and is logged as a
**v2 charter item** in [`docs/notes/v1_roadmap.md`](v1_roadmap.md).

**Reverted state**: raw blend with conditional rank-normalisation only when
scores escape [0,1] (the original v1 logic). Headline numbers return to the
v1-sealed values.

## §12 Adaptive ensemble min weight floor — REVERTED

**Initial fix**: gave non-leader models a 0.10 minimum weight floor when
the leader's adaptive score exceeded 1.25× the runner-up.

**Why reverted**: the adaptive weights are tuned against the v1
winner-take-all behavior — when fold-level scoring strongly favored one
model (LR / RF / LGBM), the v1 ensemble would commit fully to that model.
Imposing a 0.10 floor diluted strong-leader folds with under-performing
models, smoothing the ensemble's predicted distribution per-date and
collapsing OOS top-K selection: CAGR 33.3% → 18.1%, Sharpe 0.89 → 0.60,
alpha +11.7% → -2.4% (post-revert pipeline rerun 2026-04-27).

The reviewer's underlying critique stands — winner-take-all is fragile
on small numbers of OOS folds. A proper fix requires re-fitting the
adaptive scoring to expect a smoothed blend, which is a deeper change
than the inference-time patch and is logged as a **v2 charter item**.

**Reverted state**: original winner-take-all behavior restored. Headline
numbers return to v1-sealed values.

## Headline impact

After several rerun/revert cycles, the **kept fixes** (#1 sector cap,
#2 bootstrap rename, #5 turnover, #6 diagnostic columns, #7 inner-purge
config wiring, #3 default-arg disambiguation) are all backtest-only or
diagnostic. They do not touch the model training path. The final production
configuration explicitly restores the original fully invested sector
concentration behavior as `sector_mode="soft_penalty"`, while retaining the
true cash-residual cap as `sector_mode="hard_cap_cash"` for risk experiments.
The fresh headline metrics are reported in `README.md §1`.

The **reverted fixes** (#4, #8, #9, #10, #11, #12) touched the feature or
ensemble paths and individually appeared correct, but each one shifted
fold-1 LGBM AUC by 0.5-1pp, which cascaded through the adaptive
ensemble's winner-take-all threshold (1.25× leader-vs-runner-up ratio).
When the threshold no longer triggered, the ensemble blended models with
under-performing scores instead of committing to a single strong model,
compressing the predicted distribution and collapsing top-K returns
(CAGR 33% → 13-19% across various combinations).

The reviewer's underlying critiques are all valid; the proper fixes
require **re-fitting the adaptive ensemble's scoring** under the new
feature/blend assumptions, which is a deeper change than these
inference-time patches and is logged as a **v2 charter item** in
[`docs/notes/v1_roadmap.md`](v1_roadmap.md). For v1, the goal is to keep
the published headline reproducible while documenting the known
methodology gaps so the reviewer cannot legitimately surprise us in
defence.
